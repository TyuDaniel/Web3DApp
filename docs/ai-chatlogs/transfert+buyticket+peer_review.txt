**State Machine**

```text
idle
  -> unlocking        user selected file + password and clicked Unlock
  -> error            malformed file / missing password

unlocking
  -> unlocked         decrypt succeeds and initial chain reads succeed
  -> error            wrong password / malformed keystore / RPC read failure

unlocked
  -> estimating       user clicks Buy Ticket
  -> unlocking        user changes file/password and unlocks another wallet
  -> error            refresh/read failure

estimating
  -> pending          gas estimate succeeds, balance covers price + gas, signed tx sent
  -> error            sold out/revert/balance too low/estimate failure

pending
  -> success          receipt received
  -> error            sendSignedTransaction throws before receipt
  -> pending          update elapsed seconds every second

success
  -> unlocked         refresh balances completes
  -> estimating       user buys again after refresh

error
  -> unlocking        user retries unlock
  -> estimating       user retries buy while wallet remains unlocked
  -> idle             user clears keystore
```

**Error Mapping**

This is the core mapper I’d put in [js/buy.js](./js/buy.js:1):

```js
function mapBuyError(error, context = {}) {
  const message = String(
    error && (error.message || error.reason || error.data || error)
  );

  if (context.phase === 'parse-keystore') {
    return "this doesn't look like a valid keystore file";
  }

  if (context.phase === 'decrypt') {
    return 'incorrect password for this keystore';
  }

  if (message.includes('TicketToken: incorrect payment')) {
    return 'the ticket price changed — please refresh and try again';
  }

  if (message.includes('TicketToken: no tickets available')) {
    return 'the event is sold out';
  }

  if (message.includes('TicketToken: reentrant call')) {
    return 'transaction rejected by safety check — please try again';
  }

  if (context.phase === 'balance-check') {
    return `you need at least ${context.requiredEth} SETH (price + gas estimate). your balance is ${context.balanceEth}.`;
  }

  if (context.phase === 'broadcast') {
    return 'transaction may or may not have been sent. check etherscan for your address before retrying.';
  }

  return 'something went wrong. please refresh and try again.';
}
```

I’ll probably use ASCII apostrophes in the actual file for consistency unless you want the exact curly punctuation from the rubric.

**Unlock Pattern**

Recommendation: `file input + FileReader`.

Why: this matches the project’s `create-wallet` flow, where the user downloads a JSON keystore file. A textarea paste path is more error-prone, looks less like a wallet workflow, and makes it easier for the user to accidentally paste the wrong thing. File input also makes malformed JSON handling clearer.

Pattern:

```js
function readKeystoreFile(file) {
  return new Promise((resolve, reject) => {
    const reader = new FileReader();

    reader.onload = () => {
      try {
        resolve(JSON.parse(reader.result));
      } catch (error) {
        reject(error);
      }
    };

    reader.onerror = () => reject(reader.error);
    reader.readAsText(file);
  });
}
```

Then:

```js
const keystore = await readKeystoreFile(file);
const account = web3.eth.accounts.decrypt(keystore, password);
```

**Transaction Pattern**

The write flow in `buy.js` would be:

```js
const data = contract.methods.buyTicket().encodeABI();

const gas = await contract.methods.buyTicket().estimateGas({
  from: unlockedAccount.address,
  value: ticketPriceWei,
});

const gasPrice = await web3.eth.getGasPrice();
const nonce = await web3.eth.getTransactionCount(unlockedAccount.address, 'pending');

const tx = {
  from: unlockedAccount.address,
  to: CONTRACT_ADDRESS,
  value: ticketPriceWei,
  data,
  gas,
  gasPrice,
  nonce,
  chainId: 11155111,
};

const signed = await unlockedAccount.signTransaction(tx);
const receipt = await web3.eth.sendSignedTransaction(signed.rawTransaction);
```

This follows the Web3 signed transaction flow: build transaction, sign locally, broadcast signed raw transaction. The official Web3 docs describe this same sign-then-`sendSignedTransaction` model for raw transactions: [Web3.js transactions guide](https://docs.web3js.org/guides/transactions/transactions/).

**Deviations From Andy’s Pattern**

Compared with a basic `sendPayableTransaction` tutorial pattern, I’d intentionally add:

- Contract method call data: `contract.methods.buyTicket().encodeABI()` instead of a plain ETH transfer.
- `value = ticketPriceWei` read from the contract, not typed manually by the user.
- `estimateGas()` before signing so we can catch reverts and calculate required balance.
- Explicit balance check: `ticketPrice + gas * gasPrice`.
- In-page state rendering separate from toast messages.
- Persistent pending timer while waiting for receipt.
- Etherscan receipt link on success.
- Keystore upload from the generated wallet file, instead of hardcoded/private-key input.
- Error mapping for the rubric’s contract reverts and broadcast ambiguity.

**Where `transfer.js` Diverges From `buy.js`**

Most structure stays the same: file input, `FileReader`, local decrypt, state rendering, pending timer, receipt link, `beforeunload`, `CHAIN_ID`.

Actual differences:

- `latestTicketPriceWei` is removed.
- `latestTicketRaw` or `latestTicketCount` is added so the UI and defensive check know whether the user has at least one ticket.
- `refreshWalletState()` reads:
  ```js
  web3.eth.getBalance(unlockedAccount.address)
  contract.methods.balanceOf(unlockedAccount.address).call()
  ```
  and displays `DEPLOYER_ADDRESS` truncated.
- Transfer call data becomes:
  ```js
  contract.methods.transfer(DEPLOYER_ADDRESS, ONE_TICKET.toString()).encodeABI()
  ```
- Gas estimate becomes:
  ```js
  contract.methods.transfer(DEPLOYER_ADDRESS, ONE_TICKET.toString()).estimateGas({
    from: unlockedAccount.address,
  })
  ```
- Transaction object has no `value`.
- Required balance check is only:
  ```js
  gas * gasPrice
  ```
  not `ticketPrice + gas`.
- Button enablement also depends on `ticketsHeld >= 1`.
- Before signing, code repeats:
  ```js
  if (latestTicketCount < 1n) { ... }
  ```
- Success state says transfer confirmed, not purchase confirmed.

**Error Mapper**

```js
function mapTransferError(error, context = {}) {
  const message = String(
    error && (error.message || error.reason || error.data || error)
  );

  if (context.phase === 'parse-keystore') {
    return "this doesn't look like a valid keystore";
  }

  if (context.phase === 'decrypt') {
    return 'incorrect password for this keystore';
  }

  if (message.includes('ERC20: transfer amount exceeds balance')) {
    return "you don't hold a ticket to transfer";
  }

  if (context.phase === 'gas-balance-check') {
    return `you need SETH to cover the transaction fee. your balance is ${context.balanceEth}.`;
  }

  if (context.phase === 'broadcast') {
    return 'transaction may or may not have been sent. check etherscan for your address before retrying.';
  }

  console.error('Unhandled transfer ticket error:', error);
  return 'unexpected error. check the browser console for details';
}
```

**Deviation List**

Compared with a generic signed ERC-20 transfer tutorial, this page intentionally differs in these ways:

- Recipient is fixed to `DEPLOYER_ADDRESS` from `config.js`; the user cannot edit it.
- Amount is fixed to `1 * 10^18`; the user cannot transfer partial tickets or multiple tickets.
- The page shows the vendor address truncated before transfer so the destination is visible.
- The transfer button is disabled when ticket balance is below one full token.
- The code still checks `tickets >= 1` immediately before signing, even though the button should already prevent it.
- The transaction is non-payable, so there is no `value` field and the balance check only covers gas.
- `CHAIN_ID` comes from `config.js`, same as the buy page.
- Pending state gets the same `beforeunload` warning.
- The decrypted account remains in memory while the page is open, with the same security-tradeoff comment as `buy.js`.
